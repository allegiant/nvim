# Database Guidelines

> Persistent data, lockfiles, and registry conventions for this Neovim configuration.

---

## Overview

This project has no database, ORM, migrations, tables, or application queries. Treat this guide as persistence guidance for configuration state that is committed to the repo.

The only repository-level persistent state currently observed is:

- `lazy-lock.json` — plugin versions pinned by lazy.nvim.
- `.luarc.json` — lua-language-server workspace settings and known globals.
- `.vsnip/*.json` — snippet definitions.
- `lua/overseer/template/user/*.lua` — local task template definitions.

Neovim runtime state under `stdpath("data")`, `stdpath("state")`, and `stdpath("cache")` is intentionally not committed.

---

## Query Patterns

There are no DB queries. For filesystem/runtime checks, use Neovim/libuv APIs already present in the codebase:

- `vim.fn.globpath(..., false, true)` and `vim.fn.readfile()` in `lua/core/autocmds.lua` for ShaDa temp-file cleanup.
- `(vim.uv or vim.loop).fs_stat(lazypath)` in `lua/config/lazy.lua` for lazy.nvim bootstrap.
- `vim.loop.fs_realpath(".prettierrc.js")` in `lua/plugins/conform.lua` for formatter activation conditions.

Keep these checks local to the feature or plugin that needs them; do not create a repository-wide persistence abstraction unless multiple real call sites need it.

---

## Migrations

There are no migrations.

When plugin versions change, let lazy.nvim update `lazy-lock.json`; do not hand-edit commit hashes unless intentionally repairing a lockfile. Existing lock entries use plugin names as top-level keys and include `branch` plus `commit`, for example `lazy-lock.json` pins `nvim-lspconfig`, `snacks.nvim`, and `blink.cmp`.

When adding Lua workspace globals, update `.luarc.json` only for globals that really exist in the config/runtime. Current examples include `vim`, `Snacks`, and test globals such as `describe`.

---

## Naming Conventions

- Keep lockfile keys as plugin repository names generated by lazy.nvim (`bufferline.nvim`, `nvim-tree.lua`, `toggleterm.nvim`).
- Keep snippet files grouped by filetype under `.vsnip/`, as shown by `.vsnip/vue.json`.
- Keep Overseer user templates under `lua/overseer/template/user/` and name templates after the task they run, such as `lua_run.lua`.

---

## Common Mistakes

- Do not introduce database concepts, migrations, or ORM files into this repository; this is a Neovim config.
- Do not commit local Neovim data/cache output from `~/.local/share/nvim`, `~/.cache/nvim`, or platform equivalents.
- Do not bypass lazy.nvim's lockfile format when updating plugin pins.
